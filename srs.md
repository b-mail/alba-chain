# 알바체인 (Alba-Chain)

> **한 줄 소개**
> 사장님에겐 대충 찍어 올리면 끝나는 매장 자동화를, 알바생에겐 캡처 한 장으로 추가 수당까지 챙겨주는 **AI 온톨로지 기반 올인원 알바 매니지먼트 SaaS**

**핵심 가치** — 비정형 데이터(사진·캡처·텍스트)를 VLM 으로 분석하여 **[근로계약 - 업무 - 근태 - 급여]** 를 하나의 온톨로지(지식 구조)로 연결하고 자동화한다.

---

## 1. 문제 정의

### 사장님 (B2B)

- **계약 귀찮음** : 근로계약서 양식을 맞추고 일일이 작성하기 번거로움.
- **근태 관리 파편화** : 수기 장부나 카톡으로 출퇴근을 확인하는 게 비효율적임.
- **정산/송금 스트레스** : 수당 계산, 세금(3.3% 등) 계산이 머리 아프고 송금도 일일이 해야 함.

### 알바생 (B2C)

- **급여 불확실성** : 이번 달 주휴수당·세금을 떼고 난 실수령액을 미리 알기 어려움.
- **소통 부담 & 권익 불안** : 대타/추가근무 요청을 카톡으로 받았을 때, 나중에 급여에 잘 반영될지 불안하고 매번 확인하기 눈치 보임.
- **업무 누락** : 매장 To-do list 나 레시피를 제때 확인하기 어렵고 실수하기 쉬움.
- **행정 편의** : 계좌 변경이나 근로계약서 확인을 눈치 보지 않고 앱에서 바로 하고 싶음.

## 2. 핵심 기능 명세

### 2.1 근로계약서 Agent

**유저플로우 1 — 기존 알바 등록**

1. 사장님이 근로계약서를 업로드한다.
2. 근로계약서를 PDF → txt 변환한다.
3. txt 에서 근로계약 정보를 추출한다.
4. 추출한 정보를 DB 에 저장한다.

**유저플로우 2 — 신규 알바 등록**

1. 알바 정보를 줄글로 입력한다.
2. 줄글에서 근로계약 정보를 추출한다.
3. 근로계약서를 생성한다.
4. 알바에게 전송한다.

### 2.2 대타/추가근무 Agent

> 바쁠 때 대타 근무를 하거나 추가근무를 하면 증빙하기 어렵다.

**유저플로우 1 — 대타근무(substitute) 등록**

1. 대타 근무한 증빙 자료(개인톡 캡처)를 업로드한다.
2. VLM Engine 이 **[대타 근무 요청 사건]** 의 맥락과 시간을 해석한다.
3. 시스템이 직원의 기존 근로계약 온톨로지 노드 중 **매핑 후보**를 제시한다.
4. 사장(또는 알바)이 후보 중 하나에 **강제 매핑** 하고, 사장이 최종 승인하면 근태에 반영 + 실시간 예상 급여(주휴수당, 세금 3.3% 자동 차감) 그래프를 업데이트한다.

**유저플로우 2 — 추가근무(overtime) 등록**

1. 정규 근무 외 추가로 일한 증빙(카톡 캡처/텍스트)을 업로드한다.
2. VLM Engine 이 추가근무 시작/종료 시각을 해석한다.
3. 매핑 후보 제시 → 강제 매핑 → 사장 승인 시 **가산율(연장 1.5배)** 을 적용해 급여를 재계산한다.

#### 상태 머신 (status)

```
evidence_uploaded ──(증빙 업로드)──▶ interpreting ──(VLM 해석 완료)──▶ interpreted
       interpreted ──(강제 매핑)──▶ mapped ──(사장 승인)──▶ confirmed
   interpreted/mapped ──(반려)──▶ rejected
```

| status | 의미 |
| --- | --- |
| `evidence_uploaded` | 세션 생성됨, 증빙 대기 |
| `interpreting` | VLM 해석 잡 진행 중 |
| `interpreted` | VLM 해석 완료, 매핑 후보 산출됨 — 매핑 대기 |
| `mapped` | 특정 계약 노드에 매핑 확정됨 — 사장 승인 대기 |
| `confirmed` | 사장 승인 → 근태 반영 + 급여 재계산 완료 |
| `rejected` | 증빙/해석이 부정확하여 반려됨 |

#### 매핑 후보 (mapping candidates)

`interpreted` 이후 `GET /extra-shifts/{id}` 는 직원의 **조건이 확정된 근로계약**을 바탕으로 매핑 후보 목록을 반환한다.

- 후보 산출 기준: 직원의 confirmed 계약 + 해석된 근무일(요일)이 계약 스케줄과 일치하는지, 근무 시간대 인접성.
- 각 후보는 `contract_id`, `node_id`(근태 온톨로지 노드), `label`, `score`(0~1), `reason`, VLM 해석 기반 `suggested_start_at`/`suggested_end_at` 를 포함한다.
- 사장은 후보를 선택해 `PATCH /extra-shifts/{id}/mapping` 으로 **강제 매핑**한다. 시간이 생략되면 VLM 해석값으로 채운다.

#### 급여 반영 규칙

- `substitute`: 통상 시급으로 계산.
- `overtime`: 통상 시급 × **1.5**(연장 가산) 으로 계산.
- 매핑(`mapped`) 시 **잠정 예상 급여**를, 승인(`confirmed`) 시 **확정 예상 급여**를 SSE 로 push 한다.
